## 📅 26/02/2026 – Specific Feedback & Consolidation Actions

### Strengths Identified

- Clear communication of the value proposition and project focus.
- Correct identification of the main technological needs for the solution.

### Weaknesses Identified

- Team culture: the work was organised by departments, which led to an uneven distribution of coding tasks across the team.
- Sprint definition: Sprint 3 lacked visibility and, in general, all Sprints (1–3) need deeper, better detailed planning.
- Technical research: the integration and capabilities of the Google Maps API were not analysed in sufficient depth.
- Metrics and methodology: the current metrics do not define thresholds (minimum/maximum) and are not clearly linked to milestones.
- Design and UX: mockups are too flat and do not cover sufficiently detailed use cases nor error management (e.g. how we handle user frustration when something fails).
- Cost analysis: the section is unclear and incomplete; it lacks licence/tool analysis, detailed API/token costs and explicit consideration of time as a cost factor.


### Consolidation Actions Taken

- **Design reinforcement:** extend and refine mockups to include richer user flows and document how the system responds to user errors and frustration scenarios.
- **Task restructuring:** redistribute programming tasks so that all subgroups contribute code and avoid rigid "department" separation.
- **Sprint re-planning:** deepen the planning of Sprints 1, 2 and 3 to improve visibility of workload, dependencies and realistic delivery dates.
- **Metrics refinement:** define objective- vs-time based control metrics to better track team performance and detect deviations early.
- **Technical research:** expand the analysis of the Google Maps API, documenting integration options and use cases relevant to the product.
- **Cost analysis enhancement:** refine the cost section to include licence/tool analysis, detailed API and token costs, and the temporal distribution of expenses vs. execution of tasks.
