## General feedback from evaluation session

### 📅 25/03/2026 – General observations across groups

**Positive patterns observed**

- Adapting presentation branding to match the app's visual identity creates a more coherent and professional impression.
- A logical slide flow with well-connected sections improves comprehension.
- A clear and well-structured roadmap helps the audience understand the product direction.
- Recording demos in advance ensures a smoother and more reliable presentation.
- Visual consistency throughout the presentation reinforces the product's identity.
- Introducing the killer opener in a way that immediately conveys the app concept is highly effective.
- Offering multiple user pilot profiles and incentives adds credibility to the validation process.
- Separating planning columns (e.g. adding a quality check column) can improve process visibility, as long as it does not become a bottleneck.
- A good provider analysis demonstrates technical and economic rigour.
- Using a formula-based approach to measure team performance is an interesting and scalable idea.
- Expressing retrospective problems clearly and honestly is valuable for collective learning.
- Defining team roles explicitly and in detail strengthens organisational credibility.

**Areas for improvement observed**

- Presentations should spend more time on mockups to better illustrate the product.
- Revision metrics should be more restrictive and push the team towards higher quality standards.
- Sprint workload was too sequential; tasks should be parallelised to allow concurrent development.
- Economic analysis must go beyond general expenditure: separate production costs, salaries, taxes, fixed costs, variable costs, direct and indirect costs — and make it more visual.
- User pilot lists should be smaller and more curated.
- Commitment agreements should be tracked weekly and include concrete measures.
- The problem statement should be explicitly linked to identified risks.
- User pilot complaint handling should follow a clear priority list.
- Cost analysis should include amortisation tables.
- Data used in presentations should be realistic and reflect actual figures.
- Tasks should be parallelised more effectively to avoid serial bottlenecks.

**For the next sprint**

- Improve parallelisation of tasks in planning.
- Make economic analysis more detailed, structured, and visual.
- Tighten user pilot groups and formalise feedback collection.
- Link risks directly to the problem statement.
- Apply stricter quality metrics with measurable targets.

## Consolidation actions that have been carried out

### 📅 12/03/2026 – Consolidation Actions

- **Visual improvement:** added clearer visual support to better illustrate real-world problems and use cases.
- **Metrics clarification:** updated presentation to explicitly show whether metrics are being met.
- **Time tracking:** incorporated graphs showing total and per-sprint hours (Clockify).
- **User feedback integration:** clarified how user feedback is incorporated into planning.
- **Cost restructuring:** shifted focus from personal costs to fixed and operational costs.
- **Slide enhancement:** improved visual quality of cost-related slides.
- **Testing consistency:** unified testing approach for local and cloud environments.
- **Testing documentation:** expanded explanation of testing strategy aligned with the testing plan.
- **Content optimisation:** reduced or removed less relevant slides (roadmap, retrospective).
- **Team dynamics:** improved or removed ineffective team-building content and reinforced motivational elements.
